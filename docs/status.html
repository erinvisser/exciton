Branch Status & TALYS Replication

The exciton project has four branches, each representing a different stage of development toward replicating TALYS pre-equilibrium calculations. This page documents what each branch does, how it interfaces with MARLEY, what has been validated, and what remains unresolved.

2. Git History & Branch Relationships

3. MatrixSquared — The Validated Reference (preeqmode=1)

3.1 Purpose & Structure

The MatrixSquared branch implements TALYS pre-equilibrium mode 1 (preeqmode=1), which uses a constant matrix-element approximation (Kalbach-Cline parametrization) for the residual two-body interaction. It was created by commit 854f5ed ("Stripped OMP features, returning to simple matrix squared state") and preserved as a clean, minimal implementation.

Design principle: Include only the code required to accurately replicate TALYS preeqmode=1. No OMP infrastructure, no CollisionKernel enum, no wompfac, no MassTable lookups beyond what the integrator needs internally.

3.2 MARLEY Dependencies

Minimal — the only MARLEY header used is:

// transition_rates.cpp
#include "marley/Integrator.hh"

The marley::Integrator provides the Clenshaw-Curtis adaptive quadrature for the numerical integration over intermediate excitation energy. The Makefile still links all MARLEY .cc files (except marley.cc and marsum.cc), but none of the OMP or MassTable routines are ever called.

4. OMPfromScratch — OMP Kernel Rebuilt (preeqmode=3)

4.1 Purpose & Structure

The OMPfromScratch branch implements the same OMP collision kernel as OMP (preeqmode=3) but was rebuilt from the validated MatrixSquared codebase. Instead of grafting OMP code onto the early development branch, it forks from the clean M² reference and adds the OMP infrastructure incrementally:

  • KD model integration — KoningDelarocheOpticalModel instantiation, setIncidentEnergyAndFragment(), wvolRadialIntegral()
  • Wompfac formulas — same-type, cross-type, and conversion wompfac from TALYS matrix.f90
  • MassTable separation energies — Sp, Sn via MARLEY MassTable singleton
  • Guard bounds on conversion integrals — Pauli-prescribed minimum energies enforced for both created and annihilated configurations
  • Pauli correction edge cases — proper handling of j=4 density parameter and conversion guards
  • n=1 analytical locking — forces analytical formula for n=1 states even in numerical mode
  • Midpoint bin separation — different bin counts for creation vs conversion terms

5.4 Hypothesized Root Causes (common to both OMP and OMPfromScratch)

The OMP transition rate validation has never succeeded. After the KD03 parameters, well depths, and radial potential were all verified against TALYS, the rates themselves still disagreed by 30–100%. The git history reveals a trail of attempted fixes:

Issue 1 — Compound vs target nucleus (evidence: 4 reverting commits):
  • 4b20de8 "separation energy for mother, not daughter"
  • 1d654b6 "tweaking where i call target vs compound nucleus details"
  • 01f42c7 "changing kd_omp to compound nuc" → reverted by
  • 8cce1f1 "back to target..." → then
  • fa714e2 "nevermind talys omp does use target"
The KD model was instantiated with different (Z,A) pairs across commits. TALYS uses the target nucleus for the OMP evaluation, but the compound nucleus is the correct choice for the separation energy calculation. This confusion likely meant that different parts of the code used inconsistent reference nuclei.
Issue 2 — R-ratio in wompfac (evidence: commit 447cca0): Commit 447cca0 is titled "Rpinu fix". The wompfac formulas (same-type, cross-type, conversion) depend on the ratio Rπν/Rππ. If this ratio is applied incorrectly, the relative strengths of same-type vs cross-type pair creation would be wrong. The TALYS matrix.f90 source may handle the sub-channel bookkeeping differently than the implementation.
Issue 3 — Separation energy sign/definition: The kinetic energy of the colliding nucleon in the integrand is: \[ e_{\text{kin}} = e - S_{p/n}(Z_{\text{comp}}, N_{\text{comp}}) \] where Z_comp and N_comp are the compound nucleus after emission. The MARLEY MassTable provides separation energies via get_fragment_separation_energy(), which may return a different sign or reference convention than TALYS expects. Commit 65c3870 adjusted "separation energy for mother, not daughter."
Issue 4 — Hole-channel density ratio: In the OMP integrand, hole-type sub-channels multiply the collision rate by ωhole/ωparticle. The exact formula from TALYS bonetti.f90 may differ from what was implemented. Commit dc5f9e4 "updating wvolradialintegral to more closely match bonetti.f90" suggests the radial integral itself was revised, but the density ratio handling may not have been fully corrected.
Issue 5 — Single KD instance per integration call: The code creates one KD model instance per integration call and updates it at each energy point. TALYS may handle the internal state of the KD model differently (e.g., resetting intermediate cache values). This would produce systematic offsets rather than random errors.
Cumulative effect: Issues 1–5 compound. Even if each individually contributes a 10–30% error, their combination could easily produce the observed 30–100% discrepancies. A systematic audit of the integrand — step-by-step comparison with TALYS output at each intermediate value — would be needed to isolate the dominant cause.

7. Open Questions for OMP Rates

  1. What is the exact TALYS bonetti.f90 integrand? — A direct line-by-line comparison of the OMP integrand with the TALYS source would resolve whether the integrand structure, density ratios, and bounds handling are correct.
  2. Is the KD model stateful between calls? — If MARLEY's KoningDelarocheOpticalModel caches intermediate results, calling setIncidentEnergyAndFragment() repeatedly may not recompute the parameters correctly. A fresh instance per energy point would eliminate this concern.
  3. What is the correct reference nucleus for separation energies? — The back-and-forth in commits between target and compound suggests this was never definitively resolved. TALYS's own convention needs to be verified against the MARLEY MassTable API.
  4. Does the wompfac formula match TALYS matrix.f90? — The constant COMP = 0.55 may be a tunable parameter in TALYS that depends on the version. The exact formula for combining the wompfac with the OMP collision probability should be verified.
  5. Is the hole-channel density ratio correct? — The ratio ωhole/ωparticle may need to be evaluated at the same energy e, or at different energies, depending on the TALYS implementation.
  6. What integration method does TALYS use for OMP? — If TALYS uses a different radial grid (different dr, different number of bins, or adaptive quadrature) for the volume-averaged W(r), this could contribute systematic differences.
Recommended approach for debugging OMP rates: Rather than comparing only the final 84 transition rates, instrument the code to print all intermediate values (Wv, Wd, ⟨W⟩, wompfac, each integrand term, each sub-integral) and compare these individually against TALYS output. This isolates which part of the calculation chain first diverges.
